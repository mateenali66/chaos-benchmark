# Chaos Engineering Benchmark

Reproducibility package for the paper:

> **Machine learning for chaos engineering: a taxonomy and empirical benchmark of fault injection tools on microservices**
>
> Mateen Ali Anjum, Phono Technologies Inc., Kitchener, ON, Canada
>
> Submitted to *Software: Practice and Experience* (Wiley), 2026

This repository contains the infrastructure code, experiment definitions, orchestration scripts, raw data, and statistical analysis used to conduct an empirical comparison of **Chaos Mesh 2.8.1** and **LitmusChaos 3.26.0** on a [DeathStarBench](https://github.com/delimitrou/DeathStarBench) Social Network microservices testbed running on AWS EKS.

## Key Results

120 experiments (12 fault scenarios x 2 tools x 5 repetitions) were executed on a 3-node SPOT EKS cluster. Summary findings:

| Metric | Chaos Mesh | LitmusChaos | Difference |
|--------|------------|-------------|------------|
| Mean throughput (rps) | 32.9 +/- 4.6 | 19.5 +/- 1.4 | +68.7% |
| p99 latency (ms) | 384.4 | 419.3 | -8.3% (lower is better) |
| Mean error rate (%) | 0.91 | 1.45 | -37.0% (lower is better) |
| Large effect sizes (Cliff's d > 0.474) | 8/12 scenarios | -- | -- |

## Experimental Setup

### Infrastructure

- **Cluster**: AWS EKS 1.31, `us-east-1`, 3x SPOT instances (m5.xlarge / m5a.xlarge / m4.xlarge)
- **Application**: DeathStarBench Social Network (27 microservices, 962 users in social graph)
- **Monitoring**: Prometheus (kube-prometheus-stack) + Grafana + Jaeger

### Fault Scenarios

| ID | Scenario | Category | Target Service |
|----|----------|----------|----------------|
| P1 | Pod Kill | Pod/Container | compose-post-service |
| P2 | Container Kill | Pod/Container | nginx-thrift |
| P3 | Pod Failure | Pod/Container | user-service |
| N1 | Network Latency 50ms | Network | social-graph-service |
| N2 | Network Latency 100ms | Network | post-storage-service |
| N3 | Network Latency 300ms | Network | user-timeline-service |
| N4 | Packet Loss 5% | Network | home-timeline-service |
| N5 | Network Partition | Network | media-service |
| R1 | CPU Stress 80% | Resource | compose-post-service |
| R2 | Memory Pressure 80% | Resource | post-storage-service |
| A1 | HTTP Abort 503 | Application | nginx-thrift |
| A2 | gRPC Unavailable | Application | user-service |

### Protocol

Each experiment follows a four-phase protocol:

1. **Baseline** (5 min) -- steady-state metrics under load (wrk2 at 200 rps)
2. **Fault injection** (2 min) -- apply chaos experiment via `kubectl`
3. **Recovery** (1 min) -- remove fault, measure time to recover
4. **Cooldown** (1 min) -- ensure stability before next run

Load is generated by [wrk2](https://github.com/giltene/wrk2) running a mixed workload: 40% compose post, 30% read home timeline, 20% read user timeline, 10% follow user.

## Reproducing the Experiments

### Prerequisites

- AWS account with permissions for EKS, VPC, EC2, S3, ECR
- AWS CLI configured with a named profile
- Terraform >= 1.5
- kubectl
- Helm 3
- Python 3.10+
- Docker (for building the wrk2 image)

### 1. Provision infrastructure

```bash
# Create terraform.tfvars with your values:
# region, cluster_name, s3_bucket_name, cluster_admin_arns

cd terraform
terraform init
terraform apply

# Bootstrap the cluster (Helm charts, kubeconfig, namespaces)
cd ..
./scripts/setup.sh
```

### 2. Deploy the application and chaos tools

```bash
# Deploy DeathStarBench, Litmus RBAC/CRDs, initialize social graph
./scripts/post-deploy.sh

# Validate everything works (13 checks)
./scripts/smoke-test.sh
```

### 3. Build the load generator

```bash
./scripts/build-wrk2-image.sh
```

### 4. Run experiments

```bash
# Single experiment (~9 min)
./scripts/run-experiment.sh --tool chaos-mesh --scenario p1 --run 1

# All 120 experiments (~22 hours, with resume support)
./scripts/run-all-experiments.sh
```

### 5. Analyze results

```bash
cd analysis
python3 analyze.py    # Summary stats, statistical tests
python3 figures.py    # Generate all figures
python3 tables.py     # Generate LaTeX tables
```

### 6. Tear down

```bash
./scripts/teardown.sh
```

## Data Format

Each experiment produces a JSON file in `data/{tool}/{scenario}/run-{N}.json` containing:

- **wrk2 output**: throughput (rps), latency percentiles (p50, p95, p99), error counts, timeouts
- **Prometheus metrics**: per-phase (baseline, fault, recovery) CPU usage, memory usage, pod restarts, network I/O
- **Metadata**: tool, scenario, run number, timestamps, protocol timing

## Statistical Methods

- **Wilcoxon signed-rank test** (non-parametric, paired) for tool comparison
- **Cliff's delta** for effect size (small/medium/large)
- **Bonferroni correction** for multiple comparisons (36 tests)
- **95% confidence intervals** computed via t-distribution
